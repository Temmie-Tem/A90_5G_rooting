## Summary

V2443 ran the exact-gated M1 Android/Magisk ACDB payload-capture route using the V2442
runner. The V2442 wiring fix worked: after the Magisk `service.sh` activation reboot,
`run_post_module_reboot_settle()` ran at the correct boundary and observed Magisk root
ready before logcat/playback.

The run reached the intended Android playback and artifact collection window, then rolled
back to V2321 successfully. It still captured zero `/dev/msm_audio_cal` ioctl payloads
because the helper processes exited immediately with usage text.

The new wall is a helper/service option mismatch:

- `service.sh` starts with `A90_M1_SERVICE_BEGIN duration=180 max_bytes=512`.
- It starts helpers for target processes `795` and `918`.
- `helper-795.log` and `helper-918.log` contain only the helper usage line.
- Source inspection shows the helper accepts `--duration-sec` only in range `1..120`.
- The module passes `--duration-sec "$remaining"`; with the default M1 duration of `180`,
  `parse_options()` rejects the value and prints usage.

Therefore the M1 service and process discovery are working, but the ptrace observer never
enters its capture loop.

## Root Cause

`workspace/public/src/android/acdb_payload_capture/a90_acdb_ioctl_capture_threadset_v2423.c`
parses `--duration-sec` with maximum `120`:

```text
parse_int_arg(argv[++arg_index], 1, 120, &value)
```

The M1 `service.sh` generated by the V2429 module template defaults to `DURATION=180` and
passes `--duration-sec "$remaining"` to the helper. On first launch, `remaining=180`, so the
helper rejects the argument and exits before emitting a JSONL `start` event or attaching to
tracees.
