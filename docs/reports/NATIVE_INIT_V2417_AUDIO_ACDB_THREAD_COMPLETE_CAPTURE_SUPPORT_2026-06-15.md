# NATIVE_INIT V2417 — ACDB payload capture thread-complete M0 support

## Scope

- Unit: AUD-5D / N3-CAP follow-up after V2416.
- Goal: fix the M0 transient Android/Magisk-root observer so it attaches to all target process threads before considering an M1 temporary Magisk boot module.
- Device action: none. This is host-only source/test work.
- Safety boundary unchanged: Android-good observation only, no native calibration ioctl, no native speaker write, no persistent Magisk install, raw payload bytes remain private.

## V2416 blocker recap

V2416 proved the Android speaker ACDB edge still occurred while the capture window was active, but the helper captured zero `msm_audio_cal` ioctl entries:

- audio HAL PID had `/dev/msm_audio_cal` open.
- logcat showed the speaker calibration edge in `android.hardware.audio.service` worker TID `4198`.
- the helper attached only to process PIDs, and Linux ptrace attaches one thread at a time.

Therefore the miss is not yet a true early-payload miss. It is a thread coverage gap.

## Implementation

V2417 updates the existing V2415/V2416 capture support in two places:

1. Controller script generated by `native_audio_acdb_payload_capture_planner_v2415.py` now enumerates `/proc/<pid>/task/*` for every target audio process.
2. The Android-side C helper now separates:
   - `--pid <tid>`: thread to ptrace and intercept syscalls from.
   - `--fd-pid <tgid>`: process fd table owner used to resolve the ioctl fd to `/dev/msm_audio_cal`.

This avoids relying on `/proc/<tid>/fd` visibility while still tracing the actual worker thread that issues the ioctl. Per-thread outputs are named:

- `msm-audio-cal-ioctl-p<PID>-t<TID>.jsonl`

The controller also snapshots:

- `proc-<pid>-tasks.txt`
- `proc-<pid>-task-<tid>-comm.txt`
- the existing process maps/fd files

## Magisk decision

Do not move to M1 yet.

- M0 transient helper remains the default.
- M1 temporary Magisk boot module remains deferred until a thread-complete M0 live run still sees no ioctl entries while logcat proves ACDB happened.
- Magisk remains only a measurement delivery mechanism and is not a native-init runtime dependency.
